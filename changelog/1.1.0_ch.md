# ospf-kotlin 1.1.0

:us: [English](1.1.0.md) | :cn: 简体中文 | [README](../README_ch.md)

本版本基于 1.0.72 的最后一个快照提交 `839c60a`。

## 总览

1.1.0 是一次完整的平台级发布。除了 core 重构之外，这个版本还补齐了 math、multiarray、quantities 等基础模块，扩展了 framework 公共能力，完成了 CSP1D、BPP3D、Gantt Scheduling 等领域框架迁移，并补充了示例、基准测试与双语文档。

## 主要变化

### Core 建模与求解链路

- 旧的 `core.frontend.*` 和 `core.backend.*` 结构已经替换为 `core.model.*`、`core.variable.*`、`core.symbol.*`、`core.solver.*` 以及相关的 token/value 基础设施。
- `MetaModel` 成为对外建模入口，内部则通过 `MechanismModel`、线性 triad / 二次 tetrad 展开和求解调用形成更清晰的生命周期。
- 泛型数值 API 已经成为主入口。对外建模使用 `LinearMetaModel<V>` 和 `QuadraticMetaModel<V>`，并显式传入 `IntoValue<V>`；求解结果则优先暴露 `objValue`、`possibleBestObjValue`、`bestBoundValue`。
- 兼容字段和桥接适配器仍保留在兼容层，但新代码应优先走泛型路径。
- 符号与不等式工具已经迁移到 `math.symbol.*`，旧的 `solver(metaModel)` 直接求解路径以及旧版 `SlackFunction` 兼容入口不再作为默认示例流的一部分。

### 新基础模块

- `ospf-kotlin-multiarray` 提供多维数组、视图、块稀疏存储和表格结构。
- `ospf-kotlin-math` 提供代数、数值类型、几何、符号表达式、算子以及解析 / 序列化辅助。
- `ospf-kotlin-quantities` 提供物理量、单位、量纲分析和 `Duration` 互操作。
- 这三个模块现在是 core、framework 以及下游领域代码的底层基础。

### Framework 公共能力

- `ospf-kotlin-framework` 现在对外说明并提供求解器抽象、组合求解器编排、列生成与 Benders 接口、管线分层、影子价格提取、持久化表达式 DSL、网络工具、日志上下文、心跳跟踪和远程求解支持。
- framework 及其子模块的 README 已经补充为双语文档。
- 根构建同时纳入了 example 和 benchmark 模块，工具链也升级到了新的 Kotlin 与 kotlinx/http4k 版本。

### CSP1D 框架

- `ospf-kotlin-framework-csp1d` 已经从单体式应用模型，整理成可复用的一维分切框架。
- 领域被拆分为 material、切割方案生成、长度分配、yield、浪费最小化和 produce 等上下文。
- 对外应用层围绕 `Csp1dProblem`、`Csp1dSolution`、`Csp1dMilp`、`Csp1dColumnGeneration`、`Csp1dRecovery`、`Csp1dProduceContext`、`Csp1dShadowPriceLifecycle`、`Csp1dSolutionEnrichment`、`TopKCuttingPlans` 及其相关 policy/config 类型展开。
- 列生成现在会保留 LP master、原地加列、通过 `CGPipeline` 刷新影子价格，并支持通过扩展管线注入业务约束。
- 恢复与 warm start 现在支持原生初始赋值，并保留 partial solution 语义。
- 生成器的缓存与剪枝已经覆盖宽度、长度、刀数、数量、切片模板复用、跨 worker 去重和 dominance 过滤，同时保持稳定的 KPI 与 render 输出。
- 现有的 acceptance tests、Gurobi 烟测和 benchmark 快照已经覆盖完整生命周期。

### BPP3D 框架

- `ospf-kotlin-framework-bpp3d` 已经迁移到更明确的形状感知与物理量感知领域模型。
- 固定 `FltX` 别名、Quantity 兼容层、`BoundingCuboid` 兼容映射以及其他旧快捷入口已经删除，调用方应直接使用形状多态和 solver 多态 API。
- Cuboid 与 cylinder 现在通过统一的形状契约守护几何语义。横向圆柱使用原生 `HorizontalCylinderX` / `HorizontalCylinderZ` 算法形状，不再通过兼容映射表达。
- 连续半径与 PWL 半径建模已经收敛到 `ContinuousRadiusModelComponent`，并通过 `MetaModel` -> `MechanismModel` 生命周期注册 core `UnivariateLinearPiecewiseFunction` 符号。
- Renderer 与 solver info 会携带 actual volume、选中半径诊断、PWL 模型规模信息、阻断注册原因和 axis-aware cylinder 元数据。
- 边界脚本与验证用例会阻止固定数值别名、已删除兼容快捷入口和应用层自定义 PWL Big-M 注册逻辑回流。

### Gantt Scheduling 框架

- `ospf-kotlin-framework-gantt-scheduling` 已经对齐 framework 的泛型数值与物理量方向。
- 时间窗口、时隙约束、容量、资源、产出/消耗、生产率日历和结果边界优先使用 `Quantity<V>` 或明确的物理量类型。
- 废弃的 `Flt64` 时间窗口工厂、适配别名和桥接/委托链已经删除，或收敛到兼容边界。
- 对外文档现在说明了 context / aggregation / pipeline 架构、branch-and-price 生命周期、列生成 master/pricing 分层、泛型数值边界和物理量边界。
- KDoc、import 与 suppression 作用域已经整理，便于审计迁移后的 API 面。

### 示例与基准

- `ospf-kotlin-example` 作为对外 demo 和验证模块加入根工程。
- `ospf-kotlin-benchmark` 作为 JMH 基准与趋势对比模块加入根工程。

## 迁移指南

### 1. 更新构建

- 下游项目请先切换到 `1.1.0`。
- 如果你直接依赖新的基础能力，请补上 `ospf-kotlin-multiarray`、`ospf-kotlin-math`、`ospf-kotlin-quantities`。
- 如果你要使用示例或性能基准，再按需加入 `ospf-kotlin-example` 和 `ospf-kotlin-benchmark`。

### 2. 迁移 core 包路径

- 将 `core.frontend.*` 替换为 `core.model.*`、`core.variable.*`、`core.symbol.*`。
- 将 `core.backend.*` 替换为 `core.solver.*`。
- 把表达式和不等式相关代码迁移到 `math.symbol.*`。
- 将 `math.symbol.adapter.*` 替换为 `math.symbol.operation.*`。
- 不再依赖旧的 `solver(metaModel)` 直接求解兼容路径，也不要继续使用旧版 `SlackFunction` 入口。

### 3. 切换到泛型数值 API

- 建模优先使用 `LinearMetaModel<V>` 和 `QuadraticMetaModel<V>`，并显式传入 `IntoValue<V>`。
- 求解输出优先读取 `objValue`、`possibleBestObjValue`、`bestBoundValue`。
- `obj`、`possibleBestObj`、`bestBound` 只作为兼容字段看待。
- 数值类型映射建议统一使用标准 converter：
  - `Flt64` -> `IntoValue.Identity`
  - `FltX` -> `FltX.toIntoValue()`
  - `Rtn64` -> `Rtn64.toIntoValue()`
  - `RtnX` -> `RtnX.toIntoValue()`

### 4. 调整 math 与 quantities 使用方式

- 代数、几何、符号表达式和数值类型请使用 `ospf-kotlin-math`。
- 多维数组、张量式数据结构请使用 `ospf-kotlin-multiarray`。
- 物理量、单位、量纲安全和 `Duration` 互操作请使用 `ospf-kotlin-quantities`。
- 默认代码路径里不要再保留 `utils.math.*` 旧导入，也不要把无单位裸值和量纲值混着算。

### 5. 调整 framework 集成

- 以 `Pipeline`、`CGPipeline`、`HAPipeline` 作为主要扩展点。
- 业务专属约束和定价规则应放进扩展管线或 policy，不要堆进 application solver 主流程里。
- 使用 `Csp1dSolveConfig<V>.extensions` 和 `extensionSet` 注入领域行为。
- 影子价格请走 `refresh` / `extractor` 生命周期，不要手写双值回填逻辑。

### 6. 调整 CSP1D 调用方

- 用 `Csp1dProblem<V>` 和 `Csp1dSolveConfig<V>` 构造问题与配置。
- 直接 MILP 场景用 `Csp1dMilp`，价格驱动场景用 `Csp1dColumnGeneration`。
- 需要 warm start 或失败回退时，使用 `Csp1dRecovery` / `Csp1dColumnGenerationRecovery`。
- 输出增强请继续通过 `Csp1dSolutionEnrichment` 和 KPI/render 扩展点保留。

### 7. 调整 BPP3D 调用方

- 将已删除的固定 `FltX`、物料装载、层/深度限制和 `BoundingCuboid` 兼容别名替换为原生形状与物理量 API。
- 圆柱输入请使用 `PackageShapeSpec` 与 axis-aware cylinder 元数据。横向圆柱必须满足受守护的地面或 cuboid 支撑覆盖契约。
- 连续半径逻辑应保留在 `ContinuousRadiusModelComponent` 和 core PWL 符号生命周期里，不要在 application 层注册自定义 Big-M 约束。
- 涉及 cylinder 或 PWL 半径路径时，请从 actual volume 与选中半径字段读取 renderer/KPI 数据。

### 8. 调整 Gantt Scheduling 调用方

- 将废弃的 `Flt64` 时间窗口工厂和桥接别名替换为泛型 `V : RealNumber<V>` API。
- 时间、持续时长、容量、资源数量、产出数量和生产率值应使用 `Quantity<V>` 或明确的物理量类型存储。
- 下游排程规则应通过 context、aggregation、pipeline 或 limit 扩展点注入，不要直接修改 application 编排。
- `Flt64`、`Double` 和无单位裸值应限制在 solver adapter、模型注册或结果提取边界内。

### 9. 重新跑验证

- 执行默认迁移门禁：

```powershell
pwsh.exe -NoProfile -ExecutionPolicy Bypass -File .\ospf-kotlin-core\scripts\check-migration-compat.ps1
```

- 如果本地具备 solver 集成环境，再补跑 `-WithSolverIntegration` 的可选门禁。
- 重新运行 `ospf-kotlin-example` 里的泛型示例验证。
- 涉及 CSP1D 的改动，请再跑 `ospf-kotlin-framework-csp1d/README.md` 里列出的聚焦测试和烟测。
- 涉及 BPP3D 的改动，请再跑 `ospf-kotlin-framework-bpp3d/README.md` 里列出的聚焦测试、Gurobi 数据集检查、renderer schema fixture 和边界脚本。
- 涉及 Gantt Scheduling 的改动，请再跑 `ospf-kotlin-framework-gantt-scheduling/README.md` 里的本地验证命令。

## 兼容性说明

- 默认源码集里不应再出现 `src/non-default-main` 或 `src/non-default-test`。
- `core.frontend.*`、`core.backend.*`、`utils.math.*`、`math.symbol.adapter.*` 这类旧导入不应再回流。
- `FunctionCompat`、`MetaModelFlt64Adapter`、旧的 `solver(metaModel)` 以及旧版 `SlackFunction` 兼容调用都已经从默认示例路径中移除。
- BPP3D 中已经删除的兼容别名、`BoundingCuboid` 映射、自定义 PWL Big-M 注册逻辑和固定 `FltX` 别名不应回流。
- Gantt Scheduling 中废弃的 `Flt64` 时间窗口工厂、适配别名和 Quantity 桥接链不应回流。
