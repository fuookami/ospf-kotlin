# ospf-kotlin 1.1.0

:us: English | :cn: [简体中文](1.1.0_ch.md) | [README](../README.md)

This release is based on the 1.0.72 snapshot at commit `839c60a`.

## Summary

1.1.0 is a full platform release. The core rewrite is paired with math, multiarray, and quantities foundations split out from `ospf-kotlin-utils`; a larger framework surface; reusable CSP1D, BPP3D, and Gantt Scheduling framework updates; example and benchmark modules; and refreshed bilingual documentation.

## Major Changes

### Core model and solver stack

- The old `core.frontend.*` and `core.backend.*` layout has been replaced by `core.model.*`, `core.variable.*`, `core.symbol.*`, `core.solver.*`, and the related token/value infrastructure.
- `MetaModel` is the public modeling entry. `MechanismModel`, flattening to linear triad / quadratic tetrad form, and solver invocation now form a clearer internal lifecycle.
- Generic numeric APIs are now first-class. Public modeling entry points use `LinearMetaModel<V>` and `QuadraticMetaModel<V>` with explicit `IntoValue<V>` conversion, and solver outputs expose `objValue`, `possibleBestObjValue`, and `bestBoundValue`.
- Legacy compatibility fields and adapters remain only for backward compatibility; new code should stay on the generic path.
- Symbol and inequality helpers now live under `math.symbol.*`, and the old compatibility route through `solver(metaModel)` and the old `SlackFunction` path is no longer part of the default example flow.

### Foundation modules split from utils

- `ospf-kotlin-multiarray` has been split out from `ospf-kotlin-utils` and now owns multidimensional arrays, views, sparse/block storage, and tabular data helpers.
- `ospf-kotlin-math` has been split out from `ospf-kotlin-utils` and now owns algebra, numeric types, geometry, symbolic expressions, operators, and parsing/serde helpers.
- `ospf-kotlin-quantities` has been split out from `ospf-kotlin-utils` and now owns physical quantities, units, dimensional analysis, and duration interop.
- `ospf-kotlin-utils` remains the base utility module, while these three dedicated modules form the math, data-structure, and physical-quantity foundation for core, framework, and downstream domain code.

### Framework surface

- `ospf-kotlin-framework` now documents and exposes solver abstractions, combinatorial solver orchestration, column-generation and Benders interfaces, pipeline hierarchies, shadow-price extraction, persistence-expression DSL, network utilities, log context, heartbeat tracking, and remote solver support.
- Bilingual READMEs have been expanded across the framework and domain modules.
- Build metadata now includes the example and benchmark modules, and the toolchain has been moved to newer Kotlin and kotlinx/http4k versions.

### CSP1D framework

- `ospf-kotlin-framework-csp1d` has been rebuilt as a reusable one-dimensional cutting-stock framework instead of a monolithic application module.
- The domain is split into material, cutting-plan generation, length-assignment, yield, waste-minimization, and produce contexts.
- The public application surface now centers on `Csp1dProblem`, `Csp1dSolution`, `Csp1dMilp`, `Csp1dColumnGeneration`, `Csp1dRecovery`, `Csp1dProduceContext`, `Csp1dShadowPriceLifecycle`, `Csp1dSolutionEnrichment`, `TopKCuttingPlans`, and the related policy/configuration types.
- Column generation now keeps the LP master alive, adds columns in place, refreshes shadow prices through `CGPipeline`, and supports extension pipelines for business-specific constraints.
- Recovery and warm start flows now support native initial assignment values and partial-solution preservation.
- Generator caching and pruning now cover width, length, knife count, quantity, slice-template reuse, cross-worker deduplication, and dominance filtering with stable KPI and render output.
- Acceptance tests, Gurobi smoke tests, and benchmark snapshots cover the full lifecycle.

### BPP3D framework

- `ospf-kotlin-framework-bpp3d` has been migrated toward shape-aware and quantity-aware domain modeling.
- Fixed `FltX` aliases, quantity compatibility layers, `BoundingCuboid` compatibility mapping, and other stale shortcuts were removed so callers use shape-polymorphic and solver-polymorphic APIs directly.
- Cuboid and cylinder geometry now share guarded shape contracts. Native `HorizontalCylinderX` / `HorizontalCylinderZ` algorithm shapes are used for horizontal cylinders instead of compatibility mapping.
- Continuous-radius and PWL radius modeling were moved behind `ContinuousRadiusModelComponent`, which registers core `UnivariateLinearPiecewiseFunction` symbols through the `MetaModel` -> `MechanismModel` lifecycle.
- Renderer and solver info now carry actual-volume semantics, selected-radius diagnostics, PWL model-scale information, blocked registration reasons, and axis-aware cylinder metadata.
- Boundary scripts and verification suites guard against fixed-number alias reflux, deleted compatibility shortcuts, and custom PWL Big-M registration returning to application code.

### Gantt Scheduling framework

- `ospf-kotlin-framework-gantt-scheduling` has been aligned with the generic numeric and physical-quantity direction used by the rest of the framework.
- Time windows, slot constraints, capacities, resources, production/consumption, productivity calendars, and result boundaries now prefer `Quantity<V>` or explicit physical quantity types.
- Deprecated `Flt64` time-window factories, adapter aliases, and bridge/delegate chains were removed or narrowed to compatibility boundaries.
- Public documentation now describes the context / aggregation / pipeline architecture, branch-and-price lifecycle, column-generation master/pricing split, generic numeric boundary, and quantity boundary.
- KDoc, imports, and suppression scopes were cleaned up to keep the migrated API surface easier to audit.

### Examples and benchmarks

- `ospf-kotlin-example` is added as the public demo and verification module.
- `ospf-kotlin-benchmark` is added for JMH-based baseline and trend comparison work.

## Migration Guide

### 1. Update the build

- Bump downstream projects to `1.1.0`.
- Add the split-out foundation modules when you previously depended on the corresponding utilities through `ospf-kotlin-utils`, or when you depend on them directly: `ospf-kotlin-multiarray`, `ospf-kotlin-math`, `ospf-kotlin-quantities`.
- If you use the demo or performance harness, include `ospf-kotlin-example` and `ospf-kotlin-benchmark` as needed.

### 2. Move core code to the new package layout

- Replace `core.frontend.*` with `core.model.*`, `core.variable.*`, and `core.symbol.*`.
- Replace `core.backend.*` with `core.solver.*`.
- Move expression and inequality code to `math.symbol.*`.
- Replace `math.symbol.adapter.*` with `math.symbol.operation.*`.
- Stop relying on the old direct `solver(metaModel)` compatibility path or the old `SlackFunction`-style entry points.

### 3. Switch to the generic numeric API

- Prefer `LinearMetaModel<V>` and `QuadraticMetaModel<V>` with an explicit `IntoValue<V>`.
- Read solver output from `objValue`, `possibleBestObjValue`, and `bestBoundValue`.
- Treat `obj`, `possibleBestObj`, and `bestBound` as compatibility-only fields.
- When mapping number types, use the standard converters:
  - `Flt64` -> `IntoValue.Identity`
  - `FltX` -> `FltX.toIntoValue()`
  - `Rtn64` -> `Rtn64.toIntoValue()`
  - `RtnX` -> `RtnX.toIntoValue()`

### 4. Move math and quantity code

- Use `ospf-kotlin-math` for algebra, geometry, symbolic expressions, and number types.
- Use `ospf-kotlin-multiarray` for multi-dimensional array and tensor-like data structures.
- Use `ospf-kotlin-quantities` for unit-aware values, dimension checks, and duration interop.
- Remove `utils.math.*` imports from default code paths and avoid mixing raw scalars with unit-aware values.

### 5. Update framework integrations

- Use `Pipeline`, `CGPipeline`, and `HAPipeline` as the main extension points.
- Keep business-specific constraints and pricing rules in extension pipelines or policies instead of in the application solver body.
- Use `Csp1dSolveConfig<V>.extensions` and `extensionSet` to inject domain-specific behavior.
- For shadow prices, rely on the `refresh` / `extractor` lifecycle instead of manual dual-value plumbing.

### 6. Update CSP1D callers

- Build problems with `Csp1dProblem<V>` and `Csp1dSolveConfig<V>`.
- Use `Csp1dMilp` for direct MILP solves and `Csp1dColumnGeneration` for pricing-based flows.
- Use `Csp1dRecovery` / `Csp1dColumnGenerationRecovery` for warm start and fallback behavior.
- Preserve output enrichments through `Csp1dSolutionEnrichment` and the KPI/render extension points.

### 7. Update BPP3D callers

- Replace deleted fixed-`FltX`, material-packing, layer/depth-limit, and `BoundingCuboid` compatibility aliases with the native shape and quantity APIs.
- Use `PackageShapeSpec` and axis-aware cylinder metadata for cylinder inputs. Horizontal cylinders must satisfy the guarded floor or cuboid support coverage contracts.
- Keep continuous-radius logic behind `ContinuousRadiusModelComponent` and core PWL symbols instead of registering custom Big-M constraints in application code.
- Read renderer and KPI data from actual-volume and selected-radius fields when cylinder or PWL radius paths are involved.

### 8. Update Gantt Scheduling callers

- Replace deprecated `Flt64` time-window factories and bridge aliases with generic `V : RealNumber<V>` APIs.
- Store time, duration, capacity, resource amount, production amount, and productivity values as `Quantity<V>` or explicit quantity types.
- Inject downstream scheduling rules through context, aggregation, pipeline, or limit extension points rather than editing application orchestration.
- Keep raw `Flt64`, `Double`, and untyped values inside solver adapter, model registration, or result extraction boundaries.

### 9. Re-run validation

- Run the default migration gate:

```powershell
pwsh.exe -NoProfile -ExecutionPolicy Bypass -File .\ospf-kotlin-core\scripts\check-migration-compat.ps1
```

- If you have solver integration available, run the optional solver-gated mode with `-WithSolverIntegration`.
- Re-run the generic demo verification in `ospf-kotlin-example`.
- For CSP1D-specific changes, re-run the focused tests and smoke checks described in `ospf-kotlin-framework-csp1d/README.md`.
- For BPP3D-specific changes, re-run the BPP3D focused tests, Gurobi dataset checks, renderer-schema fixtures, and boundary scripts described in `ospf-kotlin-framework-bpp3d/README.md`.
- For Gantt Scheduling-specific changes, re-run the local validation command in `ospf-kotlin-framework-gantt-scheduling/README.md`.

## Compatibility Notes

- The compatibility-free default source sets should not contain `src/non-default-main` or `src/non-default-test`.
- Legacy imports such as `core.frontend.*`, `core.backend.*`, `utils.math.*`, and `math.symbol.adapter.*` should not return.
- `FunctionCompat`, `MetaModelFlt64Adapter`, and the old direct `solver(metaModel)` and `SlackFunction` compatibility calls are removed from the default example path.
- Deleted BPP3D compatibility aliases, `BoundingCuboid` mapping, custom PWL Big-M registration, and fixed-`FltX` aliases should not return.
- Deprecated Gantt Scheduling `Flt64` time-window factories, adapter aliases, and quantity bridge chains should not return.
